\begin{abstract}
Multi-head Latent Attention (MLA) compresses KV cache via low-rank latent projections, dramatically reducing memory traffic for the attention kernel.  We show that this compression creates a \emph{new} dominant bottleneck: the \textbf{reconstruction GEMMs}---weight-absorbed batch matrix multiplications that recover full-dimensional K/V from compressed latents---consume \textbf{61\% of attention-layer time} at batch size~1 on DeepSeek-V3-scale architectures (128~heads, 61~layers), exceeding the attention kernel itself.  These BMMs are memory-bound at all practical batch sizes (arithmetic intensity 1--93 vs.\ H100 roofline crossover at~295).

We investigate INT4 weight-only quantization as a mitigation.  Selective INT4 of reconstruction weights preserves quality ($\Delta$PPL~=~+0.051 on wikitext-2, vs.\ +6.057 for naive whole-model INT4).  However, a custom batched W4A16 Triton kernel achieves only 0.49$\times$ of cuBLAS FP16---we identify \textbf{L2 cache residency} as the root cause: the 16~MB reconstruction weight matrix fits within H100's 50~MB L2, invalidating the HBM-bound roofline assumption that motivates quantization.  This L2 cache barrier---where MLA's compression paradoxically makes reconstruction weights \emph{too small} to benefit from reduced precision---is, to our knowledge, the first characterization of this failure mode.

We validate our profiling methodology through an independent FlashInfer vs.\ Triton kernel comparison on Llama-3-8B, reproducing known results (89\% vs.\ 80\% HBM utilization in decode; 2.6$\times$ prefill gap from TMA hardware loads) and contributing new NCU-level root cause analysis.  All code and data are publicly available.
\end{abstract}

\section{Introduction}

Multi-head Latent Attention (MLA)~\cite{deepseekv2} has emerged as the dominant attention architecture for frontier LLMs, powering DeepSeek-V2 and V3~\cite{deepseekv3}.  By compressing the KV cache through low-rank latent projections, MLA achieves a 7.1$\times$ reduction in KV memory traffic compared to standard multi-head attention---dramatically accelerating the attention kernel.

However, this compression comes at a cost that has received little scrutiny: during inference, full-dimensional K and V must be \emph{reconstructed} from the compressed latents via weight-absorbed batch matrix multiplications (BMMs).  We show that these reconstruction GEMMs---not the attention kernel itself---are the \textbf{dominant bottleneck} in MLA attention layers, consuming 61\% of per-layer time at batch size~1 on DeepSeek-V3-scale architectures.

A natural mitigation is INT4 weight-only quantization of the reconstruction matrices.  We find that selective INT4 preserves model quality ($\Delta$PPL~=~+0.051), but a custom Triton W4A16 kernel achieves only 0.49$\times$ of cuBLAS FP16---not the 3.9$\times$ predicted by roofline analysis.  We trace this to a phenomenon we term the \textbf{L2 cache barrier}: MLA's compression makes the reconstruction weight matrices small enough (16~MB) to fit in H100's 50~MB L2 cache, invalidating the HBM-bound assumption that motivates weight quantization.

To validate our profiling methodology, we conduct an independent FlashInfer~\cite{flashinfer} vs.\ Triton~\cite{triton} attention kernel comparison on Llama-3-8B, reproducing known performance gaps and contributing new NCU-level root cause analysis (TMA hardware loads, cooperative grid launch, the occupancy paradox).  This establishes the measurement infrastructure used for our MLA analysis.

\textbf{Contributions:}
\begin{itemize}[leftmargin=*,topsep=0pt]
  \item \textbf{First quantification of MLA reconstruction GEMM overhead}: 61\% of attention-layer time at bs=1, 2.17~ms per token across 61 layers---a bottleneck invisible to standard attention benchmarks.
  \item \textbf{The L2 cache barrier}: identification and characterization of a failure mode where MLA's compression paradoxically makes reconstruction weights \emph{too small} to benefit from reduced precision, with roofline-predicted 3.9$\times$ speedup collapsing to 0.49$\times$ measured.
  \item \textbf{Selective INT4 quality safety}: reconstruction weights are uniquely quantization-friendly (+0.051 PPL), establishing them as priority targets for future kernel optimization.
  \item \textbf{NCU-level FlashInfer vs.\ Triton analysis}: TMA vs.\ global-load distinction, corrected L1 metric interpretation on Hopper, and the occupancy paradox (fewer warps, more bandwidth).
\end{itemize}

\subsection{Methodology}

\textbf{Timing.}  CUDA events with 10+ warmup iterations, median of 100+ timed iterations.  Reproducibility verified across 3 independent trials (Section~\ref{sec:repro}).

\textbf{FLOP counting.}  Standard attention: $4 \cdot B \cdot \frac{S^2}{2} \cdot H \cdot d$ (QK$^\top$ matmul + AV matmul, $\times 2$ for multiply-accumulate, $\div 2$ for causal masking).

\textbf{Bandwidth formula.}  Decode: $\text{BW} = (\text{KV}_\text{read} + Q_\text{read} + O_\text{write}) / t_\text{median}$, where KV accounts for 99.8\% of bytes.

\textbf{NCU profiling.}  \texttt{ncu --set full --profile-from-start no} with kernel replay.  Metrics collected: \texttt{dram\_\_throughput}, \texttt{sm\_\_throughput}, \texttt{sm\_\_warps\_active}, L1/L2 sector hit/miss counts, \texttt{launch\_\_registers\_per\_thread}, \texttt{smsp\_\_inst\_executed\_pipe\_tensor}.  NCU profiling overhead is excluded from bandwidth computations (separate timing runs).

\subsubsection{Finding: TMA vs.\ Global Loads (Not ``Cache Hit Rate'')}
\label{sec:tma}

A naive reading of NCU's L1 sector counters yields: FlashInfer 97\% L1 hit rate, Triton 0.1\%.  \textbf{This is misleading.}

FlashInfer's Hopper kernel uses TMA (Tensor Memory Accelerator)~\cite{hopper}---a dedicated hardware unit that performs asynchronous bulk tensor copies directly from HBM/L2 into shared memory, \emph{bypassing the L1 global load data path entirely}.  The 108K L1 global load sectors are residual metadata accesses (indptr arrays, scalar parameters), not QKV tensor data.  The 37.8M sectors in Triton represent the \emph{actual} QKV working set flowing through L1 via standard \texttt{ld.global} instructions.

The correct comparison is at L2: FlashInfer achieves 86.4\% hit rate vs.\ Triton's 72.0\%.  FlashInfer's cooperative grid launch (132 blocks = 1 per SM) creates a controlled L2 working set; Triton's 2048-block grid generates 83\% more L2 misses (9.9M vs.\ 5.4M sectors) due to wave-quantization thrashing.

This distinction is architecturally fundamental:  TMA access is not ``better caching''---it is a \emph{different hardware data path} unavailable to Triton's compiler, which generates standard \texttt{ld.global} PTX.

\subsubsection{Finding: No Register Spills (Occupancy-Only Bottleneck)}
\label{sec:spills}

Triton's prefill kernel compiles to 255 registers---the \texttt{sm\_90} hardware maximum.  A natural hypothesis is that registers spill to local memory, adding latency.  We tested this at three levels:

\begin{enumerate}[leftmargin=*,topsep=0pt]
  \item \textbf{NCU local memory counters}: \texttt{l1tex\_\_data\_pipe\_lsu\_wavefronts\_mem\_local\_op\_\{ld,st\}.sum} returned \texttt{n/a} on all Hopper kernels (known NCU limitation on sm\_90).
  \item \textbf{PTX inspection}: Zero \texttt{.local} directives, zero \texttt{st.local} / \texttt{ld.local} instructions in the compiled PTX.
  \item \textbf{CUBIN resource dump} (\texttt{cuobjdump --dump-resource-usage}): \texttt{LOCAL:0} for \texttt{\_fwd\_kernel}.
\end{enumerate}

\textbf{Verdict}: Triton's 255-register kernel does \emph{not} spill.  The Triton compiler successfully fits all live variables within the register file at the cost of maximal register allocation.  The performance impact is \emph{purely occupancy}: fewer warps per SM means fewer opportunities to hide memory latency.

This implies that reducing Triton's register count (e.g., via different tiling or explicit \texttt{num\_stages} tuning) could improve occupancy and partially close the prefill gap---but the TMA access pattern difference (Section~\ref{sec:tma}) would remain.

\subsubsection{Finding: Cooperative Grid vs.\ Wave Quantization}

FlashInfer launches exactly 132 thread blocks---one per SM---using CUDA cooperative launch semantics.  This eliminates wave-quantization effects and enables persistent-kernel-style execution where each block processes multiple tiles sequentially with full shared memory and register file utilization.

Triton launches 2,048 blocks ($15.5\times$ more), dispatched in multiple waves.  Each wave evicts the previous wave's L2 residency, explaining the 14-point L2 hit rate gap (86.4\% vs.\ 72.0\%).

\section{MLA Reconstruction GEMM Overhead}
\label{sec:recon}

Multi-head Latent Attention~\cite{deepseekv2} compresses KV cache to a low-rank latent $c_t \in \mathbb{R}^{d_\text{lora}}$ with $d_\text{lora}=512$.  During decode, full-dimensional K and V are \emph{reconstructed} via weight-absorbed batch matrix multiplications (BMMs):

\begin{align}
\text{BMM1 (Q absorption):} \quad & Q'_h = Q_h \cdot W^\text{kc}_h, \quad W^\text{kc}_h \in \mathbb{R}^{d_\text{nope} \times d_\text{lora}} \nonumber \\
\text{BMM2 (V reconstruction):} \quad & O_h = A_h \cdot W^\text{vc}_h, \quad W^\text{vc}_h \in \mathbb{R}^{d_\text{lora} \times d_v} \nonumber
\end{align}

where $h = 1, \ldots, H$ heads.  These are batched GEMMs with batch dimension $H$, independent of KV sequence length.

\subsection{Why Reconstruction Weights Are Quantization-Friendly}

The $W^\text{kc}$ and $W^\text{vc}$ matrices are \emph{projection weights} that map between a compressed 512-dimensional latent space and 128-dimensional head spaces.  Several factors explain their quantization robustness:

\begin{enumerate}[leftmargin=*,topsep=0pt]
  \item \textbf{Low-rank structure}: The latent space was trained to be compressible; the projection matrices inherit smooth spectral properties.
  \item \textbf{Redundancy}: With $H=128$ heads each projecting from the same 512-dim latent, per-head reconstruction errors are averaged across heads before affecting output.
  \item \textbf{Post-softmax attenuation}: BMM2 operates on attention-weighted values; quantization noise is attenuated by the softmax distribution's sparsity.
\end{enumerate}

\subsection{Theoretical Speedup}

For memory-bound operations, speedup from weight quantization equals the ratio of total data transferred:

\begin{equation}
\text{Speedup} = \frac{\text{Weight}_\text{FP16} + \text{Activation}_\text{FP16}}{\text{Weight}_\text{INT4} + \text{Scale/ZP} + \text{Activation}_\text{FP16}}
\label{eq:speedup}
\end{equation}

\subsection{The L2 Cache Barrier: Why Roofline Fails for Small-Matrix MLA Reconstruction}
\label{sec:l2barrier}

The 8$\times$ gap between roofline-predicted (3.9$\times$) and measured (0.49$\times$) performance demands explanation.  We identify three factors, the first of which is dominant:

\textbf{(1) L2 cache residency invalidates the HBM-bound assumption.}  The standard roofline model~\cite{roofline} assumes data is served from HBM at 3.35~TB/s.  But the total reconstruction weight per BMM is $128 \times 128 \times 512 \times 2 = 16$~MB---comfortably within H100's 50~MB L2 cache.  After first access, \texttt{torch.bmm} serves weights from L2 at $\sim$12~TB/s, not HBM.  Reducing weight precision from FP16 to INT4 saves HBM bandwidth that \emph{was never the bottleneck}.

This creates a paradox specific to MLA: the same low-rank compression that makes reconstruction weights small enough to be a latency concern \emph{also} makes them small enough to be L2-resident, defeating the primary motivation for weight quantization.  Standard LLM linear layers (e.g., FFN projections with weights in the hundreds of MB) do not exhibit this behavior.

\textbf{(2) INT4 dequantization overhead.}  Each packed byte requires bit masking, shifting, signed extension, and type conversion---adding compute that the roofline model does not account for.  The stride-2 activation loads required by even/odd INT4 packing further reduce cache efficiency.

\textbf{(3) cuBLAS batched dispatch advantage.}  \texttt{torch.bmm} uses hardware-optimized batched GEMM scheduling on H100 with fused warp-level batching, while our Triton kernel launches a separate thread block per head-tile combination.

\textbf{When does INT4 help?}  The L2 cache barrier is not absolute.  In production serving, concurrent operations (FFN GEMMs, attention across multiple layers, multi-tenant request batching) contend for L2 capacity, potentially evicting reconstruction weights back to HBM.  This suggests INT4 reconstruction gains are \textbf{deployment-dependent}: negligible in isolated benchmarks, but potentially significant in high-throughput serving with L2 pressure.  Alternatively, fusing reconstruction across multiple layers to exceed L2 capacity, or deploying on GPUs with smaller L2 caches, could restore the roofline-predicted benefit.

\section{Discussion}

\subsection{Implications for Compiler-Generated Attention}

The prefill gap is architecturally fundamental: Triton's MLIR$\to$PTX pipeline generates \texttt{ld.global} instructions because it has no TMA code generation path for attention patterns.  Triton 3.x introduced \texttt{tl.experimental\_descriptor\_load} for TMA access, but this requires manual descriptor setup that attention kernels have not adopted.  Closing the prefill gap requires either:

\begin{enumerate}[leftmargin=*,topsep=0pt]
  \item Triton compiler support for automatic TMA lowering of attention-shaped loads (research problem).
  \item Manual TMA integration in the Triton kernel source (engineering, loses portability).
  \item A higher-level DSL that targets both TMA and global load paths depending on the GPU architecture.
\end{enumerate}

The decode gap (1.1$\times$) is more tractable: Triton's two-phase split could be fused, and register tuning (\texttt{num\_warps}, tiling factors) could recover some bandwidth.

\subsection{The Occupancy Paradox as Design Principle}

FlashInfer's decode kernel demonstrates that for memory-bound workloads, \emph{maximizing bandwidth per warp} outperforms \emph{maximizing warp count}.  At 183 registers, FlashInfer has 2.4$\times$ fewer active warps than Triton, yet extracts 10\% more HBM bandwidth.  This is consistent with the general principle that memory-bound kernels benefit more from coalesced access patterns and prefetching than from occupancy~\cite{volkov2010}.

\subsection{MLA: Compression Shifts the Bottleneck}

MLA's 7.1$\times$ KV traffic reduction makes the attention kernel faster---but creates a new bottleneck in the reconstruction GEMMs that were previously negligible.  At bs=1, reconstruction costs 61\% of attention-layer time.  This is a classic systems pattern: optimizing one component (KV cache reads) reveals a previously hidden cost (weight-matrix reads for reconstruction).

\subsection{Selective Quantization: Quality Result}

Despite the kernel limitations, reconstruction weights are \emph{uniquely quantization-friendly}: +0.051 PPL for selective INT4 vs.\ +6.057 for naive INT4 of all weights.  This establishes a clear design principle for future MLA serving: \textbf{reconstruction weights are the first target for aggressive quantization}, pending kernel support.

\subsection{Limitations}

\begin{itemize}[leftmargin=*,topsep=0pt]
  \item Single GPU; multi-GPU tensor-parallel workloads may show different ratios due to all-reduce overlap.
  \item GPU clocks not locked; absolute numbers carry $\sim$3\% uncertainty at small batch sizes.
  \item NCU local memory counters unavailable on Hopper (\texttt{n/a}); spill analysis relies on PTX/CUBIN inspection.
  \item INT4 kernel benchmarked in Triton only; a CUDA-native implementation with INT8 tensor cores may achieve different results.
  \item Perplexity evaluated on DeepSeek-V2-Lite (15.7B); larger models may show different quantization sensitivity.
  \item One model architecture per attention type.
\end{itemize}

\section{Related Work}

\textbf{Attention kernels.}  FlashAttention~\cite{flashattention} and FlashAttention-2~\cite{flashattention2} introduced tiling-based exact attention with $O(N)$ memory.  FlashInfer~\cite{flashinfer} extends this with Hopper-native TMA support.  Triton~\cite{triton} enables Python-level GPU kernel development.  SGLang~\cite{sglang} integrates both backends for LLM serving.

\textbf{MLA architectures.}  DeepSeek-V2~\cite{deepseekv2} introduced Multi-head Latent Attention; DeepSeek-V3~\cite{deepseekv3} scaled it to 671B parameters with 128 attention heads.  While both papers describe the weight-absorbed reconstruction mechanism, neither quantifies its runtime cost relative to the attention kernel.

\textbf{Performance modeling.}  The roofline model~\cite{roofline} provides the standard framework for reasoning about compute- vs.\ memory-bound kernels.  Our L2 cache barrier finding (Section~\ref{sec:l2barrier}) demonstrates a failure mode of single-level roofline analysis: when working sets fit in L2, the effective bandwidth ceiling shifts from HBM to L2, invalidating roofline predictions.  Volkov~\cite{volkov2010} showed that occupancy is not always the primary performance factor, motivating our analysis of FlashInfer's register-heavy approach.

\textbf{Weight quantization.}  GPTQ~\cite{gptq} and AWQ~\cite{awq} enable post-training INT4 weight quantization for LLM inference.  KVQuant~\cite{kvquant} applies quantization to the KV cache itself, targeting the memory footprint that MLA addresses architecturally.  QuaRot~\cite{quarot} uses rotation to eliminate outliers before 4-bit quantization.  However, all prior quantization work targets standard linear layers (FFN, attention projections)---not the reconstruction matrices specific to MLA.  To our knowledge, no prior public work quantifies the reconstruction GEMM overhead in MLA or evaluates selective INT4 quantization of these weights.

% ══════════════════════════════════════════════════════════════════════════════
\section{Conclusion}

We presented two sets of findings.  First, a kernel-level characterization of FlashInfer vs.\ Triton attention on H100, identifying TMA hardware loads, memory access pattern quality, and cooperative grid launch as the three root causes of the performance gap.  Second, and more significantly, we identified MLA's reconstruction GEMMs as a dominant and previously unquantified bottleneck---consuming 61\% of attention-layer time at batch size 1 for DeepSeek-V3-scale architectures.

Our most nuanced finding concerns INT4 quantization of reconstruction weights.  \textbf{Quality is safe}: selective INT4 adds only +0.051 perplexity.  \textbf{But speedup is not automatic}: our custom Triton W4A16 kernel achieves 0.49$\times$ of cuBLAS FP16 because the 16~MB reconstruction weights fit in H100's 50~MB L2 cache, making HBM bandwidth savings irrelevant.  This reveals a gap between roofline-predicted (3.9$\times$) and realized performance that is specific to small, L2-resident weight matrices---a regime common in MLA but absent from standard LLM linear layers.  Closing this gap requires either CUDA-native INT8 tensor core kernels, cross-layer weight fusion beyond L2 capacity, or deployment on GPUs with smaller L2 caches.

All profiling scripts, raw data, and the perplexity evaluation code are publicly available for reproduction.
